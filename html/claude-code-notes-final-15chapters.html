Video 01 · Orientation

Why Claude Code, and the line between vibing and engineering.

Before the tooling, the mindset. The first lesson isn't a command — it's knowing what you're trying to build, and whether this is a moment to play or a moment to engineer.

What you need to bring

  • System-level understanding — you, the human, still drive the architecture.
  • Treat Claude Code as an AI coding assistant — let it write, you direct.
  • Know what you want to build. Vague intent produces vague output.

Vibe coding vs. AI-assisted engineering

These aren't the same activity. Vibe coding is exploratory and fast. AI-assisted coding is deliberate. Knowing which mode you're in changes how you use the tool.

Vibe coding shines

  • Proofs-of-concept
  • MVPs
  • Exploring ideas
  • Hackathons
  • Low-stakes projects, generally

Vibe coding fails

  • Scalable production systems
  • Critical infrastructure
  • Anything with serious money on the line
  • Code that has to be maintained for years

Why Claude Code specifically

  • Raw coding intelligence — strong at the actual work.
  • Long-context handling — holds large codebases in working memory.
  • Refactoring & architecture — thinks structurally, not just line-by-line.
  • Agentic capabilities — can plan and execute, not just suggest.
  • Behaves like a senior engineer — pushes back, asks, scopes.
Goal of the playlist

Give you a flavor of AI-assisted coding, ground you in the fundamentals of Claude Code, and eradicate the fear that comes with letting an AI touch your code.

Video 03 · The Vocabulary

Slash commands & the anatomy of a session.

Slash commands are shortcuts you type inside a Claude Code session, starting with /, that fire a predefined action without you having to write the whole prompt. They exist because repetition is the enemy of flow.

Two flavors

Built-in

  • Ship with Claude Code itself.
  • Available in every session, every project.
  • Type / to see the full list.

Custom

  • You build them yourself.
  • Live in your .claude/ folder.
  • Encode your workflows, your patterns.

What is a session?

  • A session is one conversation with Claude Code — from claude to /exit.
  • It has a unique ID, the full message history, every file read, every tool result, and the current working directory.
  • Sessions save automatically to ~/.claude/projects/ and can be resumed any time — even after closing the terminal.
  • Run claude -r to load past sessions, pick one, continue.

Good session hygiene

  • One session = one task. Don't braid features together.
  • Name your session immediately — future-you will thank you.
  • Commit frequently within a session.
  • Use /btw for quick, throwaway questions.
  • Export the session before a big refactor — insurance.

Making changes that stick.

The mechanics of getting Claude to modify real code: how you point at files, how you describe what you want, and how you keep your repo's history readable while doing it.

The workflow

  • Use @ to mention specific files in the project directory.
  • Write detailed prompts — the more context, the better the diff.
  • Draft your prompt, refine it with another LLM (ChatGPT, Gemini), then hand it to Claude.
  • If you have a reference image or mockup, paste it directly into chat and ask for matching changes.
  • Commit even small meaningful changes — small commits, big leverage.
> @src/components/Header.tsx add a sticky behavior on scroll
>   with a subtle backdrop blur. Match the style of @src/components/Footer.tsx.
>   Don't introduce new dependencies.

# multimodal: drop in an image, then —
> make the empty state look like this. keep our color tokens.
Multimodal, properly used

Claude Code accepts images directly in the conversation. Use them for visual specs, design references, error screenshots, whiteboard photos — anything words alone fail at.

Video 05 · The Constraint

The context window, and why every session is finite.

A context window is the amount of information — measured in tokens — that the model can see and remember at once while generating a response. Think of it as working memory. It's the most important constraint in the whole stack.

Context, defined broadly

Context is everything the model needs to understand what you're asking. Not just the prompt — all of it:

  • The codebase itself
  • PRDs and specs
  • JIRA tickets, GitHub issues
  • Slack threads, messages
  • Previous AI chats
  • Git history, prior PRs

Context window = the maximum slice of all that the model can hold simultaneously.

Things that are easy to forget

  • Claude Code's context window is 200k tokens (not the latest API max).
  • Each session starts fresh — nothing carries from yesterday on its own.
  • Both your messages and Claude's replies consume tokens.
  • Claude's replies are roughly 6× heavier than your messages.
  • Every request resends the entire conversation history from scratch.
  • Subagents get their own isolated context window — completely separate.
  • Subagents return only a summary, not their full working history.

Why context size actually matters

What it affects

  • Cost — every token has a price tag.
  • Workflow shape — long tasks need a plan.
  • Response quality — degrades as the window fills.

How to protect it

  • One session per feature.
  • /compact proactively, not reactively.
  • Focused, specific prompts.
  • Subagents for isolated or exploratory work.
  • .claudeignore for irrelevant files.

What happens when the window fills

Failure isn't a cliff — it's a gradient. Four stages:

01
Stage 1
Quality degrades

Replies get sloppier, more forgetful, slightly off-target.

02
Stage 2
Auto-compaction

At ~75–92% full, Claude summarizes history into the autocompact buffer.

03
Stage 3
Corruption

Repeated compaction loses fidelity. Details get rewritten or lost.

04
Stage 4
Hard stop

The buffer itself fills. Session can't continue meaningfully.

Recovering room

  • /compact — summarize the chat into the autocompact buffer (while it has space).
  • Spin up a subagent for the next task — fresh window, isolated.
  • /clear or start a new session — nuclear, but clean.
Rule of thumb

Compact before you need to. By the time quality has already degraded, the summary itself will inherit the degradation.

Video 06 · The Most Important File

Claude.md — your project's memory.

LLMs don't remember anything. Claude Code can't recall yesterday's session. Repeating yourself across sessions is cumbersome, error-prone, and leads to inconsistent generation. CLAUDE.md is the file that solves this — a persistent system prompt for your project.

What it actually is

A project-level instructions file Claude Code reads as context every time. Instead of telling Claude over and over how your code is laid out, what conventions matter, how to run the tests, what tools to use — write it down once, here.

  • How your project is structured
  • Coding conventions you follow
  • How to run, build, test
  • Which tools to use (and which to avoid)

Two ways to create it

Manual

  • You write the file by hand.
  • More effort, more accurate.
  • You decide what makes the cut.

/init

  • Claude scans your repo and drafts it.
  • Faster than reading everything yourself.
  • Spots patterns you might forget.
  • Good template to start from.

How /init works under the hood

  • Spawns an internal agent that takes over scanning and writing.
  • Reads high-signal config first: package.json, requirements.txt, Makefile, README.md.
  • Walks the directory tree.
  • Infers tech stack, layout, naming conventions.
  • Writes CLAUDE.md to the project root.
Honest math

The generated CLAUDE.md is roughly 30% useful. The other 70% — workflows, constraints, things to avoid, deployment targets, naming conventions — is on you.

What goes inside

  • Project Context — short description so Claude immediately knows what it's working on.
  • Architecture — codebase structure, where things live.
  • Code Style — conventions, formatting, idioms.
  • Constraints — required (or forbidden) tools and frameworks.
  • Commands — exact commands to run, test, lint, deploy.
  • Critical Rules — warnings, edge cases, landmines.
  • Roadmap — where this is going.
  • Things to Avoid — what's off-limits.

The .claude folder

Claude Code's local configuration directory. Holds skills, custom slash commands, sub-agents, settings — everything that shapes behavior for a project or for your whole machine.

Project-level · ./.claude/

  • Scope: this project only.
  • Lives in the repo, shared with the team.
  • Project-specific commands, workflows, settings.

Global · ~/.claude/

  • Scope: every project on your machine.
  • Personal — not in any repo.
  • Your defaults, your style, your shortcuts.
.claude/ ├── settings.json # shared, committed ├── settings.local.json # personal, gitignored ├── commands/ # custom slash commands ├── rules/ # long-form instructions ├── skills/ # reusable skill packages └── agents/ # subagent definitions

The four flavors of Claude.md

Project root

  • ./CLAUDE.md
  • The main one. Committed to the repo.

Inside .claude/

  • ./.claude/CLAUDE.md
  • Behaves identically to project-root version.

Local · gitignored

  • CLAUDE.local.md
  • Personal tweaks. Read alongside main file. Never pushed.

Global

  • ~/.claude/CLAUDE.md
  • Applies across every project on your machine.

And there's a fifth: subdirectory CLAUDE.md. Starting from the current working directory, Claude Code recurses up to / and reads any CLAUDE.md or CLAUDE.local.md it finds. Especially useful in monorepos and large codebases — keep instructions next to the code they describe.

Practices that hold up

  • Start with /init, then prune aggressively.
  • Commit it to git.
  • Only put universally applicable things in it.
  • Use emphasis sparingly — if everything is IMPORTANT, nothing is.
  • Keep it short, ideally under 200 lines. Instruction-following quality drops as line count rises.
  • Test: would removing this line cause a mistake? If no, cut it.
  • Treat it as a living document. Build it organically.
  • Correct once, then codify.
  • Audit periodically — instructions drift.

When 200 lines aren't enough

Don't just keep adding. Split it.

.claude/rules/ ├── code-style.md ├── testing.md ├── security.md └── api-conventions.md # these only load when Claude needs them

Or use @imports in your CLAUDE.md to reference external docs:

## API Conventions
See @docs/api-guidelines.md

## Git Workflow
See @docs/contributing.md

Or simply use subdirectory CLAUDE.md files next to the code they govern.

Auto memory

A persistent directory where Claude records learnings, patterns, and insights as it works. When it discovers something about your project — "this app uses INR, not USD" — it saves it. Next session, it already knows.

  • Lives at ~/.claude/projects/<project>/memory/memory.md
  • Only the first 200 lines of MEMORY.md auto-load.
  • Run /memory in any session to review or edit what's been saved.
  • You can also ask Claude directly: pass the instruction in a prompt, ask it to update memory.md.
The mental model

CLAUDE.md is what you choose to tell Claude up front. Auto memory is what Claude figures out on its own and writes down. One is curated, one is observed. Together, they make a project feel like it has continuity.

Spec-driven development — designing before generating.

Vibe coding feels productive because the keystrokes-to-output ratio is incredible. But the question isn't how fast can I get code? — it's how fast can I get the right code? Spec-driven development inverts the order. Think first, write the spec, then let Claude build to it.

The problem with prompts like "build me an auth system"

A casual prompt hands the steering wheel to the model. Every decision you didn't make becomes a decision it made — usually plausibly, sometimes wrongly, almost never the way you'd have chosen if asked.

> Build me a user auth system.

# the questions you didn't answer, but somebody has to:
#   • which framework?
#   • JWT or sessions?
#   • password rules?
#   • what happens on 3 failed attempts?
#   • email verification flow?
#   • password reset mechanism?

You get code fast. It may not be the right code. You land in a loop of corrections and patches — each one a small bargain you didn't intend to make.

The real cost

The issue with vibe coding isn't speed — it's control. The faster you generate, the further ahead the unstated assumptions get from your intent.

What spec-driven development actually is

A software development approach where a detailed specification is written before any code. The spec becomes the single source of truth for what the system should do, and all development flows from it.

What goes in the spec — varies by company and need, but typically:

  • Problem statement — what are we solving, and why.
  • Functional requirements — what the system must do.
  • API contracts — inputs, outputs, data shapes.
  • Constraints — what we can't or won't do.
  • Edge cases and error handling — the failure modes you've thought about.
  • Acceptance criteria — how we'll know it's done.

# Spec: Registration

## Overview
Implement user registration so new visitors can create a Spendly account.
This step upgrades the existing stub GET /register route into a fully
functional form that accepts a POST, validates input, hashes the password,
and inserts a new row into the users table. On success the user is
shown a success message and then redirected to the login page. This is
the entry point for all authenticated features that follow.

## Depends on
- Step 01 — Database setup (users table, get_db())

## Routes
- GET /register  — render registration form — public (already exists as stub, upgrade it)
- POST /register — process registration form, insert user, redirect to /login — public

## Database changes
No new tables or columns. The existing users table
(id, name, email, password_hash, created_at) covers all requirements.

A new DB helper must be added to database/db.py:
- create_user(name, email, password) — hashes the password with werkzeug,
  inserts a row into users, returns the new user's id.
  Raises sqlite3.IntegrityError if the email is already taken
  (UNIQUE constraint).

## Templates
- Modify: templates/register.html
  - Change the form action to url_for('register') with method="post"
  - Add name attributes to all inputs: name, email, password, confirm_password
  - Add a block to display a flash error message
    (e.g. "Email already registered", "Passwords do not match")
  - Keep all existing visual design

## Files to change
- app.py — upgrade register() to handle GET and POST; add flash + redirect logic
- database/db.py — add create_user() helper
- templates/register.html — wire up form action/method and flash message display

## Files to create
None.

## New dependencies
No new dependencies. Uses werkzeug.security (already installed) and
Flask's built-in flash / redirect / url_for.

## Rules for implementation
- No SQLAlchemy or ORMs
- Parameterised queries only — never use f-strings in SQL
- Hash passwords with werkzeug.security.generate_password_hash — never store plaintext
- app.secret_key must be set in app.py for flash() to work
  (use a hardcoded dev string for now)
- Server-side validation must check:
  1. All fields are non-empty
  2. password == confirm_password
  3. Email is not already registered (catch sqlite3.IntegrityError)
- On any validation failure, re-render the form with a flashed error message — do not redirect
- On success, flash a success message and redirect to url_for('login')
- Use abort(405) if an unsupported HTTP method reaches the route
- All templates extend base.html
- Use CSS variables — never hardcode hex values
- Use url_for() for every internal link — never hardcode URLs

## Definition of done
- [ ] GET /register renders the registration form without errors
- [ ] Submitting the form with all valid fields creates a new user in users
      and redirects to /login
- [ ] Submitting with mismatched passwords re-renders the form with an error message,
      no DB insert
- [ ] Submitting with an already-registered email re-renders the form with
      "Email already registered" error
- [ ] Submitting with any empty field re-renders the form with a validation error
- [ ] Password is stored as a hash — never plaintext — verifiable by inspecting spendly.db
- [ ] No duplicate user is created on repeated valid submissions with the same email

# Spec: Login and Logout

## Overview
This feature implements user authentication for Spendly. It converts the
/login stub into a functional POST handler that verifies credentials
against the database, stores the authenticated user's ID in the session,
and redirects to the dashboard (or a suitable landing page). It also
implements the /logout stub, which clears the session and redirects
to the landing page. After this step, the app can distinguish logged-in
users from guests, which is a prerequisite for all expense features.

## Depends on
- Step 01 — Database Setup (users table must exist)
- Step 02 — Registration (create_user and password hashing must be in place;
  a user must exist to log in against)

## Routes
- GET /login  — render login form — public
- POST /login — validate credentials, set session, redirect — public
- GET /logout — clear session, redirect to / — public (no login required to log out)

## Database changes
No database changes. The users table created in Step 01 already stores
email and password_hash.

## Templates
- Modify: templates/login.html — add a POST form with email and password
  fields, flash message display, and a link to /register

## Files to change
- app.py — implement login() as GET+POST handler and implement logout()
- database/db.py — add get_user_by_email(email) helper that returns
  a user row or None
- templates/login.html — add POST form and flash display

## Files to create
No new files.

## New dependencies
No new dependencies. werkzeug.security.check_password_hash is already
available via the existing werkzeug install.

## Rules for implementation
- No SQLAlchemy or ORMs — use raw sqlite3 via get_db()
- Parameterised queries only — never use f-strings in SQL
- Passwords verified with werkzeug.security.check_password_hash
- Session key for the logged-in user must be session["user_id"] (integer)
- Use flask.session — do not roll a custom session mechanism
- Use CSS variables — never hardcode hex values
- All templates extend base.html
- Use url_for() for every internal link — never hardcode paths
- On failed login show a generic flash error ("Invalid email or password.") —
  do not reveal which field was wrong
- After successful login redirect to url_for("landing") until a dashboard route exists

Custom slash commands — your prompts, codified.

The moment you find yourself pasting the same multi-step prompt for the third time, it should become a slash command. Custom slash commands turn your repeatable workflows into one-line shortcuts — your prompts, version-controlled, stored on disk, behaving exactly like built-ins.

What they are

  • Saved prompts you invoke with /command_name.
  • Used to execute any repeatable workflow in your project.
  • Stored as markdown files inside the .claude folder.
  • In a sentence: user-defined commands stored as files on disk.

Two scopes

Same mechanism, two homes. Where you put the file decides who can use it.

Project-scoped

  • Stored in .claude/commands/
  • Available only within that project.
  • Lives in the repo — your team shares them.
  • For project-specific workflows: spec creation, code review, seeding data.

User-scoped

  • Stored in ~/.claude/commands/
  • Available across every project on your machine.
  • Personal — never in any repo.
  • For your style: quick formatting, commit-message generation, your favorite refactors.

What people actually build

A few that show up across teams and become muscle memory fast.

/review
Run a code-review checklist on the current file.
/commit
Generate a git commit message from the staged changes.
/test
Run the test suite and analyze any failures.
/security-scan
Scan for common vulnerabilities — SQL injection, exposed credentials, unsafe deserialization.

1 Frontmatter

YAML between --- delimiters. Tells Claude what the command does, what arguments to expect, and which tools it's allowed to call.

2 Prompt body

The actual instructions Claude executes when the command is invoked. Numbered steps work well — they help Claude follow the sequence.

Filename rule

The filename — minus .md — becomes the command. my-command.md → /my-command.

$ARGUMENTS

The magic variable. Whatever you type after the command name gets captured here for Claude to parse.

Creating one, step by step

  • Create a commands/ directory inside .claude/.
  • Add a new markdown file in it — the filename will become the command name.
  • Open it and add the frontmatter: description, allowed-tools, optionally argument-hint.
  • Write the prompt below the frontmatter. Be specific. Use numbered steps.
  • Exit your Claude session and restart it. New commands are picked up on startup.
  • Type / — your new command should appear in the list.
Naming rule

The filename is the command. seed-expense.md becomes /seed-expense. Use kebab-case, keep it short, and make it descriptive enough that you'll remember what it does six months later.

Commands that take arguments

The frontmatter field argument-hint declares what your command expects. Inside the prompt body, $ARGUMENTS captures whatever was typed after the command name. Your job is to tell Claude how to parse it — and how to fail gracefully if it can't.

# frontmatter declares the shape

# prompt body parses it

  - user_id  — integer

  - months   — integer

the usage example. Do not guess defaults.
Defensive parsing

Always include a failure path. "If any argument is missing or invalid, stop and say…" — that one line saves you from a command that silently does the wrong thing on a typo.

Skills — turning Claude into a specialist.

Claude is a generalist by design — strong across many domains, deeply expert in none. Skills are how it becomes a specialist on demand, without ever leaving the generalist conversation. A skill is a folder. The folder holds expertise. Claude loads it when it's needed and forgets it when it isn't.

The problem: general capability isn't enough

Imagine you're on a marketing team. Every day you build PowerPoint decks and present them to leadership. You ask Claude for help. Claude knows roughly what to do — but the gap between general competence and reliable, on-brand output is wide enough to swallow your afternoon.

What Claude already knows

  • What PowerPoint is.
  • How to reason about slide structure.
  • Which Python library to reach for (python-pptx).

What Claude doesn't know

  • How to handle layout properly.
  • Which fonts to use, what sizes, what weight.
  • Where and how to plot charts and tables.
  • Your company's design guidelines.

The same gap appears across every domain — frontend, data analysis, document writing, code review, infra. Claude can do the work. Doing it well, the way your team would, takes context Claude doesn't have.

Why detailed prompts aren't the answer

"Just write a detailed prompt" is the obvious response. It's also the one that doesn't scale.

  • You have to remember and retype the same long prompt every time.
  • If you stuff it into a system prompt, you burn context window on every turn — even when the skill isn't relevant.
  • Prompts are awkward to bundle with files — templates, scripts, and reference docs all live elsewhere.
  • You can't share, version, or improve a prompt the way you'd manage code.
  • Prompts don't compose — you can't reference one prompt from inside another.

What skills actually are

A skill is a reusable, file-based resource that provides Claude with domain-specific expertise — workflows, context, best practices. Skills load on demand and eliminate the need to repeatedly provide the same guidance across conversations.

Every skill is a folder. The folder always contains a SKILL.md file at the top level, and optionally supporting files — scripts, references, assets — that the SKILL.md points to when relevant.

Anatomy of a skill folder

.claude/skills/my-skill/ ├── SKILL.md # required, the entry point ├── scripts/ # executable code, loaded on demand │ ├── validate.py │ └── unpack.py ├── resources/ # reference docs, loaded if needed │ └── xml-schema.md └── assets/ # templates, fonts, images for outputs ├── template.docx └── logo.png

The SKILL.md itself has the same two-part shape as a slash command:

  • YAML frontmatter with name and description — the description is what Claude reads to decide whether this skill is relevant.
  • Markdown body with the actual instructions, code patterns, pitfalls, validation steps, and pointers to other files in the folder.

The body tells Claude when to read a resource or run a script — only then are those files loaded into context. That's the trick that makes the whole system efficient.

Progressive disclosure

The core idea behind skills: don't present information until the moment it's needed. Content is staged across three levels, each with a different loading cost.

01
The description
A single-line description in the YAML frontmatter.
Visible to Claude at all times so it can decide whether the skill is relevant. Costs almost nothing.
Always loaded
02
The SKILL.md body
The full instructions, patterns, and code in the markdown body.
Loaded into context only when Claude decides this skill applies to the current task.
On demand
03
Reference resources
Scripts, templates, schemas, and reference docs in the folder.
Loaded only when the SKILL.md body explicitly points to them mid-task.
If needed
Why this matters

Without progressive disclosure, every skill you have would cost context just by existing. With it, you can have dozens of specialist skills available without any of them eating your token budget until they're called on.

Two scopes

Same mechanism as custom slash commands. Where you put the folder decides who can use it.

Personal · ~/.claude/skills/

  • Scope: every project on your machine.
  • Personal — not in any repo.
  • Your own conventions, helpers, formatting preferences.

Project · .claude/skills/

  • Scope: this project only.
  • Lives in the repo, shared with the team.
  • Project-specific expertise — design system, deployment workflow, naming conventions.

Three ways to create one

  • Manually — write the folder and SKILL.md from scratch.
  • With skill-creator — a built-in skill that helps you bootstrap a new one.
  • From community sources — adopt skills others have written and shared.

Five steps to building one

  • Identify the need — what task does Claude keep getting almost right?
  • Create the directory and write the SKILL.md.
  • Add supporting files if needed — scripts, templates, references.
  • Test the skill against real prompts.
  • Iterate. The first version is rarely the final one.

Where this is going

One detail worth knowing: there's a forthcoming change where commands and skills will merge. The separate .claude/commands/ directory goes away — what we used to call commands will live as skills, with one extra YAML flag distinguishing the two behaviors.

disable-model-invocation

Skills auto-load when Claude detects the description matches the current task. If you want a command-style skill — one that fires only when the user explicitly invokes it — add disable-model-invocation: true to its YAML frontmatter. That single line is what turns a skill back into a command.

- Surface (cards): white (#FFFFFF) with a soft border (#E5E7EB) and/or tiny shadow

- Primary accent: a single confident color — indigo/violet (#6366F1), emerald (#10B981), or similar. Pick one and stick with it.

use arbitrary values like 13px or 27px.

## What to avoid

- Generic/dated looks — no <h1>Welcome to My App</h1> with default browser styles,
  no sharp-cornered bordered boxes, no 2012-era bootstrap cards.
- Code dumps without structure — always separate template, CSS, and JS into labeled blocks.
- Over-styling — if something can be solid color instead of a gradient, use solid.
  If it can be a border instead of a shadow, use border. Restraint reads as quality.
- Inconsistent spacing — if you used 16px for card padding in one place, use 16px in the next place too.
- Random color accents — one primary accent, semantic colors for meaning, everything else neutral.
- Clever-but-unclear UX — a clearly-labeled button beats a mystery icon. In finance, trust matters more than cuteness.
- Mobile afterthought — use CSS that works at narrow widths. Stack cards vertically

Sub-agents — when Claude delegates to itself.

The trick that makes everything else work. The main agent stays small and focused; the heavy lifting — reading a codebase, drafting a plan, running an audit — happens in a fresh, isolated context window that exists only long enough to return its answer. The result lives on. The working scratch does not.

The problem: stateless LLMs

LLMs don't remember anything between turns. Ask "What's the capital of France?" and then "What about Germany?" — without context, the model genuinely doesn't know what "about" means.

The standard workaround is simple: send the entire conversation history with every new query. The application replays everything so the model can act like it remembers.

# turn 1
user> What's the capital of France?
claude> Paris.

# turn 2 — system silently resends the full history
[user: What's the capital of France?]
[claude: Paris.]
user> What about Germany?
claude> Berlin.

Why this breaks coding agents

For a chat application, that workaround is fine — a couple of sentences per turn. For a coding agent, it isn't, because the "history" includes the codebase, and the codebase has to be re-sent on every single turn. Imagine a 30k-token codebase and a feature that takes eight turns to build:

Lost in the middle

Even when a model accepts an enormous context, it tends to weight tokens at the start and end of the input more heavily than the ones in between. Bigger context isn't free quality — it's worse attention to the middle. Just stuffing more in is the wrong instinct.

What sub-agents are

A sub-agent is a specialized AI assistant that runs in its own isolated context window. The main agent calls it like a function — hand input in, get a result back. The internal work doesn't pollute the caller's context.

Same shape as calling a function in code. You don't need to know what happens inside; you only care about the return value.

Main agent
context: small
"Add auth to my Express app."
spawns isolated context
Sub-agent
context: 30k codebase, isolated
Reads. Analyzes. Drafts an implementation plan.
returns 500-token plan
Main agent
context: still small + plan
Executes the plan. Sub-agent's context is destroyed.

The sub-agent's working memory is gone the moment it returns. Only the result survives. The main agent's context window stays clean turn after turn.

Four advantages

i

Context isolation

Each sub-agent gets a completely new context window for its specific task. Nothing it reads pollutes the main session.

ii

Specialization

Sub-agents can have their own system prompts, tool allowlists, skills, permissions, and even their own model choice — a security reviewer on Opus, a research agent on Sonnet.

iii

Modularity

Different stages of the software lifecycle — exploration, planning, implementation, review, testing — become discrete agents instead of one monolithic loop.

iv

Parallelism

Independent tasks run concurrently. EDA across three datasets, three services built simultaneously — work that's serial in a single agent becomes parallel across sub-agents.

Common use cases

Six patterns that show up across teams.

Codebase exploration
The most common case. Sub-agents read large codebases without filling the main agent's context. Claude Code reaches for this automatically.
Code review
A separate, unbiased agent reviewing what another agent wrote. The author's blind spots aren't the reviewer's.
Testing
A different agent generates the tests. Same logic as code review — the code-writer's biases shouldn't shape the tests.
Multi-stage pipelines
One agent writes an API contract, another implements it, a third tests it. The hand-off between stages is the data, not the conversation history.
Parallel independent tasks
Three services in three files, three datasets needing EDA. Concurrent work that would be serial in a single agent.
Security audit
A dedicated reviewer looking only for vulnerabilities. The code-writing agent rarely catches its own footguns.

Custom sub-agents — building your own AI workers.

Last chapter introduced the idea: a sub-agent is a sandboxed worker the main agent delegates to. This chapter is about building your own. Custom sub-agents are markdown files that define a worker's identity, tools, model, and instructions. The four reference agents at the end show what that looks like in practice.

Why custom over built-in

Built-in sub-agents are generalists. They can review code, but not the way your team reviews code. They can test, but not against your conventions. Once you have a quality bar or a checklist, generic competence stops being enough.

Consider a security audit. A built-in agent will catch SQL injection, exposed API keys, the obvious things. But if your company has a security checklist — specific patterns to flag, specific risks for your stack, specific tone for the report — generic competence won't deliver that. A custom sub-agent lets you specify exactly what to look for, which tools to use, which model to spend, and how to report. The same logic applies to testing, documentation review, infrastructure changes — anywhere generic isn't good enough.

## What to Test — Coverage Checklist
For every feature, systematically cover:
1. Happy path: correct input produces correct output/redirect/template
2. Auth guard: unauthenticated requests to protected routes return 302 to /login or 401
3. Validation errors: missing fields, invalid data, duplicate entries return appropriate errors
4. DB side effects: after a write operation, query the DB to confirm the record was created/updated/deleted
5. HTTP semantics: correct status codes (200, 201, 302, 400, 404, etc.)
6. Template rendering: response contains expected HTML landmarks or text
7. Edge cases: empty strings, very long input, SQL injection attempts

## Code Quality Rules
- Use assert statements with informative messages
- Never use time.sleep() — tests must be deterministic
- Each test must be fully independent — no shared mutable state between tests
- Use pytest.mark.parametrize for data-driven tests
- Never hardcode URLs — use Flask's url_for() within an app context
- Parameterized SQL only — if you write any raw SQL in fixtures, use ? placeholders

## What You Review

Review only the recently changed or newly added code — not the entire codebase.

## Core Security Checklist (Beginner-Focused)

Focus on these four high-impact categories.

### 1. SQL Injection
The most famous web vulnerability — and the easiest to prevent.
- Queries should use parameterized queries with ? placeholders
- Watch for f-strings, .format(), or string concatenation inside SQL
- Risky: db.execute(f"SELECT * FROM users WHERE id = {user_id}")
- Safe: db.execute("SELECT * FROM users WHERE id = ?", (user_id,))

Why it matters: an attacker could type SQL into a form field and read or destroy
the database.

### 2. Authentication Basics
- Passwords should be hashed with werkzeug.security.generate_password_hash —
  never stored in plaintext
- On login, session.clear() should be called before setting new session data
- Logout should fully clear the session

### 3. Authorization (Who Can See What)
- Protected routes should check session.get('user_id') before doing anything
- Routes that take a resource ID should verify the resource belongs to the current user

### 4. Sensitive Data Exposure
- Passwords, tokens, and secrets should never appear in logs, error messages, or
  HTTP responses
- Use abort() for HTTP errors — raw string returns can leak internals
- debug=True should not be hardcoded in production paths

## Things to Mention Lightly (Not Block On)

- XSS: watch for | safe in templates on user input
- CSRF: Spendly doesn't have CSRF protection yet. Mention this once as a known
  project-wide topic worth learning about
- Input validation: good practice but mention as improvement opportunities, not failures

    🌱 Nice to have

MCP — Claude's universal connector.

Until now, Claude Code has only known the local file system and the shell. MCP — the Model Context Protocol — is what lets it reach further: into your database, your Figma board, your GitHub repos, your Jira tickets, your AWS account. One protocol, written once, that everything plugs into.

The problem MCP solves

Before MCP, connecting an LLM to an external tool meant writing custom, non-standardized glue code for that specific API. Service updates broke the connection. Every new tool meant a new integration written from scratch. It didn't scale.

MCP, introduced by Anthropic about a year and a half ago, is the standard interface for this. The protocol defines how an LLM talks to a tool. Each tool maintainer writes one MCP server. Each LLM client speaks one protocol. They connect.

What Claude Code can do without MCP

Out of the box, Claude Code's tools are limited to local operations:

  • Read — files on the local filesystem.
  • Write — create files and write into them.
  • Bash — execute command-line commands.

What it cannot do without help: read your Git repos as Git repos (not just local files), pull from Drive, fetch your Jira tickets, summarize Slack threads, talk to your database directly. MCP closes every one of those gaps.

The hidden cost

Connecting an MCP server isn't free. The descriptions of every tool that server exposes get loaded into Claude's context window when a session starts — whether you're going to use those tools today or not. Connect six servers, each with twenty tools, and you've burned a meaningful slice of your budget before the conversation begins.

Callback to Chapter 5

Recall the 200k window from Chapter 5 — system prompt, system tools, skills, messages, and the autocompact buffer all live inside it. MCP tool descriptions live there too. Every connected server is a permanent rent on your context budget, taken before the first user message. Connect lavishly and you'll feel it as degraded response quality, not just higher cost.

The discipline

  • Connect only the servers you actually use on a given project.
  • Disconnect ones you're not actively using — /mcp remove takes one line.
  • Audit your /mcp list periodically. Servers accumulate the way browser tabs do.
  • If a server's value isn't outweighing its context cost, it's costing more than it's worth.

Hooks — turning probability into certainty.

A small, sharp tool for keeping a probabilistic system honest. Most of what Claude Code does depends on the LLM following instructions — which it does, most of the time. Hooks are scripts the harness runs deterministically, at specific moments in a session, regardless of what the LLM thinks. They're how you turn usually into always.

Claude Code, viewed as a harness

Step back from the user-facing view (terminal + LLM + tools) and look at the system. Claude Code is a coding harness — a piece of software that takes the raw power of an LLM and turns it into a reliable software engineering system. The horse has the speed; the harness makes it useful.

What lives inside the harness:

  • Reading the filesystem to provide context to the LLM.
  • Displaying terminal output for the human reader.
  • Tracking conversation history across sessions.
  • Watching the context window and offering /compact or /clear.
  • Sending API requests to Anthropic; parsing tool calls back out.
  • Permission prompts before destructive operations.
  • Memory management — CLAUDE.md, sub-agent memory files, auto-memory.
  • Slash commands, sub-agents, MCP servers, plugins, hooks.

The LLM is the brain; the harness is the body and nervous system. "I should read app.py" is a thought in the brain. The harness is what opens the file, copies the content, and sends it back. The brain has the ideas. The body does the work.

The boss and the employee

Inside this design, the LLM acts as the boss and the harness as the employee. The boss issues instructions ("run this command", "edit this file"); the employee carries them out. The split matters because the two have opposite natures.

The LLM — probabilistic

  • Same prompt can yield different outputs.
  • Hallucinates, occasionally confidently wrong.
  • Stateless across calls.
  • Disconnected from the real world — only sees text.
  • Cannot act safely on its own.

The harness — deterministic

  • Same instruction, same execution, same result.
  • Faithful by design — does what it's told.
  • Has access to real systems: files, shells, networks.
  • Software, not a model. Predictable.
The 2% problem

You can write the rule in CLAUDE.md — "never delete files," "never touch .env," "always use parameterized SQL" — and the LLM will follow it most of the time. Most of the time isn't always. On the 2% of turns where context is full or attention drifts, the LLM forgets, and the obedient harness carries out a destructive command without hesitation. Hooks close that gap.

Two concepts to know first

Hooks attach to specific moments inside a session. To know which moments matter, you need two prerequisites — the agent loop and the session lifecycle.

The agent loop

One user task usually takes many round-trips. The LLM thinks, the harness acts, the result feeds back. Repeat until the task is done.

User → "Add a /delete API endpoint to my website."
  LLM:    I need to read app.py.
  Harness: opens app.py, sends content back.
  LLM:    I need schema.sql to understand the database structure.
  Harness: opens schema.sql, sends content back.
  LLM:    I have enough context now. Writing the /delete handler in app.py.
  Harness: writes the code into app.py.
  LLM:    I should test the endpoint.
  Harness: runs the tests. All pass.
  LLM:    Done.

That iterative round-trip is the agent loop. Every coding harness has one. Hooks fire at specific moments inside each turn of the loop.

The session lifecycle

Zoom out one level. The session is the lifespan of one Claude Code conversation — from claude to /exit. Inside it, agent loops run again and again, separated by user prompts. Between every transition, the harness fires a named event. Hooks attach to those names.

What hooks actually are

A hook is a custom script the harness runs automatically when one of these events fires. Same script, same logic, every time the event triggers — regardless of what the LLM thinks. That's the whole trick. CLAUDE.md is a request to the model; a hook is a guarantee from the harness.

Anatomy of a hook

Hooks are declared in .claude/settings.json. Each one has three parts.

● ● ● .claude/settings.json
{
  "hooks": [
    {
1 Event       "event": "pre_tool_use",
2 Matcher       "matcher": "bash",
3 Action       "action": "python .claude/hooks/block_dangerous.py"
    }
  ]
}
1 Event — when

Which moment in the session lifecycle triggers this hook. One of the events from the grid above.

2 Matcher — if

Narrows the trigger. A hook that prevents file deletion only matters for bash or write — never for read.

3 Action — do what

The shell command to run. Usually a script. Receives the tool call's JSON via stdin; signals success or block via exit code.

A worked example — blocking a dangerous delete

Protect the project's database file from an over-eager LLM. The hook fires before every bash command, inspects what's about to run, and exits with a non-zero code if it sees something destructive.

First, the configuration:

# .claude/settings.json
{
  "hooks": [
    {
      "event": "pre_tool_use",
      "matcher": "bash",
      "action": "python .claude/hooks/block_dangerous.py"
    }
  ]
}

Then the script the hook runs:

# .claude/hooks/block_dangerous.py
import json, sys

# the harness sends the tool call to us as JSON on stdin
data = json.loads(sys.stdin.read())
command = data.get("tool_input", {}).get("command", "")

PROTECTED = ["spendly.db", ".env", "migrations"]

for item in PROTECTED:
    if "rm" in command and item in command:
        print(f"BLOCKED: hook prevented deletion of {item}.")
        sys.exit(2)   # exit 2 → harness blocks this tool call

sys.exit(0)              # anything else → allow it

What happens when the LLM tries to rm spendly.db:

01
User says: "Clean up the project."
02
LLM decides to run rm spendly.db. Issues a bash tool call.
03
Harness sees a bash call. Looks up pre_tool_use hooks with a bash matcher. Finds ours.
04
Harness invokes block_dangerous.py, piping the tool call as JSON to stdin.
05
Script reads JSON, finds rm spendly.db, prints the block message, exits with code 2.
06
Harness receives exit 2 → cancels the tool call. The rm never runs.
07
Harness sends the block reason back to the LLM as an error.
08
LLM reads the error, course-corrects, proposes a safer cleanup that leaves spendly.db alone.
The shape of every hook

Read JSON from stdin → make a decision → exit 0 to allow, non-zero to block. Once you see this pattern once, every other hook is a variation of it. Write your own formatter hook in fifteen lines. Write a notifier in five.

The takeaway

The problem, told as a story

Meet Rahul. Senior data scientist at a fintech, building credit-risk models that decide who gets a loan. Avid Claude Code user. Six months of customization and his productivity has roughly doubled.
